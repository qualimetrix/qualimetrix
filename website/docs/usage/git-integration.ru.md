# Интеграция с Git

AI Mess Detector интегрируется с Git, позволяя анализировать только изменённый код. Это экономит время и помогает обнаружить проблемы до того, как они попадут в основную ветку.

## Зачем анализировать только изменённый код?

Запуск полного анализа на большой кодовой базе может занять время. Что ещё важнее, в legacy-проекте может быть сотни существующих нарушений, которые сейчас не ваша ответственность. Интеграция с Git решает обе проблемы:

- **Скорость** -- анализ только тех файлов, которые вы изменили
- **Фокус** -- вы видите только те нарушения, которые вы внесли
- **Постепенное внедрение** -- начните использовать AIMD, не исправляя каждую старую проблему

---

## Быстрый старт

Два самых распространённых сценария:

```bash
# Перед коммитом: проверка staged-файлов
bin/aimd analyze src/ --staged

# Перед мержем: проверка изменений относительно main
bin/aimd analyze src/ --diff=main
```

---

## Pre-commit: работа с --staged

Флаг `--staged` ограничивает анализ файлами, добавленными в staging-область Git (`git add`):

```bash
bin/aimd analyze src/ --staged
```

Это эквивалентно:

```bash
bin/aimd analyze src/ --analyze=git:staged
```

Анализируются только PHP-файлы, которые сейчас подготовлены к коммиту. Это быстро и даёт обратную связь перед коммитом.

### Автоматический pre-commit хук

Вместо того чтобы запускать `--staged` вручную, установите Git-хук:

```bash
bin/aimd hook:install
```

Это создаёт скрипт `.git/hooks/pre-commit`, который автоматически запускает AIMD на staged-файлах перед каждым коммитом. Если обнаружены нарушения с уровнем `error`, коммит блокируется.

Проверить статус хука:

```bash
bin/aimd hook:status
```

Удалить хук:

```bash
bin/aimd hook:uninstall
```

Если у вас уже был pre-commit хук, AIMD сделает его резервную копию. Чтобы восстановить его:

```bash
bin/aimd hook:uninstall --restore-backup
```

!!! warning "Внимание"
    `hook:install` не перезапишет существующий хук без флага `--force`.

## Workflow для PR с --diff

Флаг `--diff` показывает нарушения только в файлах, изменённых относительно указанной Git-ссылки:

```bash
# Сравнение с веткой main
bin/aimd analyze src/ --diff=main

# Сравнение с конкретной веткой
bin/aimd analyze src/ --diff=origin/develop

# Сравнение с конкретным коммитом
bin/aimd analyze src/ --diff=abc1234
```

Это эквивалентно:

```bash
bin/aimd analyze src/ --report=git:main..HEAD
```

!!! note "Примечание"
    При использовании `--diff` AIMD по-прежнему анализирует всю кодовую базу (полные метрики нужны для правил уровня пространства имён). Он только *фильтрует вывод*, показывая нарушения из изменённых файлов.

---

## --analyze vs --report

В AIMD есть два отдельных механизма ограничения области:

| Опция       | Управляет   | Что делает                             |
| ----------- | ----------- | -------------------------------------- |
| `--analyze` | **Входом**  | Какие файлы парсить и собирать метрики |
| `--report`  | **Выходом** | Какие нарушения показывать в отчёте    |

### --analyze

Ограничивает набор файлов, которые AIMD обрабатывает:

```bash
# Парсить и анализировать только staged-файлы
bin/aimd analyze src/ --analyze=git:staged
```

Это быстрее, так как парсится меньше файлов, но метрики уровня пространства имён и связанности могут быть неполными, так как AIMD не видит полную картину.

### --report

Анализирует все файлы, но фильтрует отчёт, показывая только нарушения из изменённых файлов:

```bash
# Анализировать всё, выводить только изменённые файлы
bin/aimd analyze src/ --report=git:main..HEAD
```

Это даёт точные метрики (анализируется вся кодовая база), но показывает только релевантные нарушения.

### Комбинирование

Можно комбинировать оба варианта для точного управления:

```bash
# Парсить только изменённые файлы, выводить только изменённые файлы
bin/aimd analyze src/ --analyze=git:main..HEAD --report=git:main..HEAD
```

### Что использовать?

| Сценарий                        | Рекомендация                          |
| ------------------------------- | ------------------------------------- |
| Pre-commit хук (важна скорость) | `--staged` (использует `--analyze`)   |
| Ревью PR (важна точность)       | `--diff=main` (использует `--report`) |
| CI-пайплайн с полным анализом   | `--report=git:main..HEAD`             |

---

## --report-strict

По умолчанию при использовании `--diff` или `--report` AIMD также показывает нарушения из родительских пространств имён изменённых файлов. Это полезно, потому что добавление класса в пространство имён может привести к превышению лимитов размера.

Если вы хотите видеть нарушения только из самих изменённых файлов:

```bash
bin/aimd analyze src/ --diff=main --report-strict
```

---

## Синтаксис областей

Оба варианта `--analyze` и `--report` принимают выражения области:

| Выражение                  | Значение                                        |
| -------------------------- | ----------------------------------------------- |
| `git:staged`               | Файлы, подготовленные к коммиту                 |
| `git:main..HEAD`           | Файлы, изменённые между main и HEAD             |
| `git:origin/develop..HEAD` | Файлы, изменённые между удалённой веткой и HEAD |
| `git:abc1234..HEAD`        | Файлы, изменённые с указанного коммита          |

## Примеры рабочих процессов

### Локальная разработка

```bash
# 1. Вносите изменения
# 2. Добавляете в staging
git add src/Service/UserService.php

# 3. Проверяете перед коммитом
bin/aimd analyze src/ --staged

# 4. Если чисто, коммитите
git commit -m "refactor: simplify UserService"
```

Или автоматизируйте с помощью хука:

```bash
# Одноразовая настройка
bin/aimd hook:install

# Теперь каждый коммит проверяется автоматически
git commit -m "refactor: simplify UserService"
# AIMD запускается автоматически, блокирует коммит при ошибках
```

### Ревью пулл-реквеста

```bash
# На feature-ветке, проверка относительно main
bin/aimd analyze src/ --diff=main

# Строгий режим: только нарушения в ваших изменённых файлах
bin/aimd analyze src/ --diff=main --report-strict

# С JSON-выводом для CI
bin/aimd analyze src/ --diff=main --format=json --no-progress
```

### CI-пайплайн (GitHub Actions)

```yaml
- name: Run AIMD
  run: bin/aimd analyze src/ --diff=origin/main --format=sarif --no-progress > results.sarif

- name: Upload SARIF
  uses: github/codeql-action/upload-sarif@v3
  with:
    sarif_file: results.sarif
```

### CI-пайплайн (GitLab CI)

```yaml
code_quality:
  script:
    - bin/aimd analyze src/ --diff=origin/main --format=gitlab --no-progress > gl-code-quality-report.json
  artifacts:
    reports:
      codequality: gl-code-quality-report.json
```
